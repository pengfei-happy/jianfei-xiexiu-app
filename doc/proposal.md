# 秦昊减脂打卡助手需求方案

## 1. 项目概述

项目名称：秦昊减脂打卡助手

项目定位：面向大基数减脂用户的离线本地打卡 App，围绕秦昊本人原版15天减脂食谱和大冰7天循环减脂方案，提供饮食计划查看、每日打卡、体重记录、运动记录和数据看板能力。

技术栈规划：

- React Native + TypeScript
- Tailwind RN
- Zustand 本地状态管理
- AsyncStorage 本地持久化
- 图表库用于周/月体重趋势展示

目录规范：

- `doc`：存放需求、方案、数据说明等项目文档
- `src`：存放页面、组件、状态、类型和业务逻辑
- `assets`：存放静态资源

第一阶段交付范围：仅输出本文档 `doc/proposal.md`，不生成设计稿、不编写业务代码、不修改食谱内容。

## 2. 核心食谱数据源说明

唯一食谱数据源：

- `/Users/man/Desktop/秦昊本人原版15天减肥食谱（6.8-6.22｜大基数166斤+每日运动版）.docx`

数据使用原则：

- 秦昊15天食谱和大冰7天循环食谱必须严格沿用原文档内容。
- 不允许自行新增、删减、替换、估算每日三餐。
- 不允许自行修改掉秤预估、饮食禁忌、运动规则和进食规则。
- 文档中未提供的数据，在 App 中标记为“待确认”或不展示，不能编造。

已确认原文档包含：

- 秦昊15天计划：前5天极速减脂期 + 后10天复食巩固期
- 大冰7天循环计划：第16天起，7天为周期，可无限循环
- 每日三餐内容
- 秦昊15天每日预计掉秤
- 通用饮食规则
- 每日固定运动规则
- 称重标准和重要须知
- 大冰循环的16:8轻断食、进食顺序和循环规则

15天总预估掉秤：11.3斤。该数字来自原文档，作为秦昊15天计划总预估展示值使用。

## 3. 用户与核心规则

目标用户：大基数减脂用户。

体重设置：

- App 不预设初始体重。
- 用户首次使用时自行填写初始体重和目标体重。
- 体重单位固定为“斤”。
- 累计减重按用户真实体重记录计算，不按预估掉秤累加。

默认计划：

- 首次进入 App 默认执行“秦昊15天计划”。
- 秦昊15天完成后，提示用户切换到“大冰7天循环计划”，不自动强制切换。
- 用户可在个人中心手动切换计划。

打卡规则：

- 每天包含体重记录、饮食打卡、运动打卡三类动作。
- 任意完成一项，即视为当日打卡成功。
- 三项都完成时，视为当日完整打卡。
- 连续打卡天数按“当日是否打卡成功”统计。
- 若自然日未完成任何一项打卡，连续打卡天数清零。
- 支持补打卡，补打卡会影响对应日期的历史记录和连续打卡统计。

数据展示口径：

- 只展示饮食和运动记录，不计算基础代谢。
- 热量缺口不做自动精确计算，除非后续确认摄入热量和运动消耗规则。
- 运动消耗热量数值暂标记为待确认。

### 4.3 大冰7天循环减脂模块

大冰计划定位：

- 秦昊15天计划结束后无缝衔接
- 7天为一周期
- 可无限循环
- 主打养护代谢、规避平台期、防反弹、精细化燃脂塑形

核心规则：

- 16:8轻断食：每日进食窗口固定10:00-18:00
- 18点后仅饮用温水、无糖茶、黑咖啡，不再进食
- 早餐吃粗不吃精
- 午餐正常七分饱
- 晚餐主打水饱无主食
- 进食顺序：先喝汤，再吃蔬菜，再吃肉，最后吃主食
- 不喝含糖饮品
- 不吃夜宵
- 不强行光盘
- 每日饮水2000ml
- 延续每日爬坡1小时 + 力量训练30分钟

循环规则：

- 第1天到第7天结束后，下一天重新从第1天开始
- 历史记录保留真实日期和循环第几天
- 体重稳定达标后，可每10天安排一次轻度放纵餐

数据看板核心卡片：

- 今日运动打卡状态
- 连续打卡天数

- 周视图：最近7天
- 月视图：最近30天
- 数据来源：用户真实录入体重
- 若一天多次记录，默认以当天最后一次记录作为图表值

- 仅在秦昊15天计划中展示
- 总目标：11.3斤
- 真实累计减重：初始体重 - 最新体重
- 完成度：真实累计减重 / 11.3斤

- 日期
- 所属计划
- 计划天数或循环天数

- 饮食打卡状态
- 运动打卡状态

- 饮食打卡完成比例：已完成饮食打卡天数 / 计划已执行天数
- 运动打卡完成比例：已完成运动打卡天数 / 计划已执行天数

## 5. 打卡规则

每日打卡状态：

- `none`：当天未完成任何记录
- `partial`：当天完成体重、饮食、运动任意一项
- `complete`：当天体重、饮食、运动三项全部完成

当日打卡成功：

- 体重记录完成，或
- 饮食打卡完成，或
- 运动打卡完成

连续打卡：

- 从最近连续的成功打卡日期向前统计。
- 若某自然日无任何打卡，连续天数清零。
- 补打卡后需要重新计算连续打卡天数。

饮食打卡：

- 支持“按食谱完成”和“未按食谱完成”。
- 支持填写违规进食记录。
- 违规进食字段包括：日期、食物名称、备注。
- 若选择“未按食谱完成”，当天仍可算作饮食记录已填写，但不计入饮食完成率。

运动打卡：

- 爬坡快走和力量训练可分别记录。
- 完成任意运动项即视为当天有运动记录。
- 两项都完成时，计入运动完整完成。

## 6. 数据计算逻辑

体重相关：

- 初始体重：用户填写
- 最新体重：用户最近一次记录
- 累计减重：初始体重 - 最新体重
- 单日变化：当日体重 - 上一次有记录的体重
- 单位：斤

计划天数：

- 秦昊计划：固定15天
- 大冰计划：7天循环
- 大冰循环日计算：`((执行天数 - 1) % 7) + 1`

秦昊预估掉秤：

- 使用原文档每日预计掉秤。
- 总预估掉秤固定为11.3斤。

大冰预估掉秤：

- 原文档未提供每日具体掉秤数字。
- App 不自行估算。
- 展示“大冰循环以稳步减脂、维稳塑形为主”。

热量相关：

- 当前版本不自动计算摄入热量。
- 当前版本不自动计算运动消耗热量。
- 当前版本不自动计算热量缺口。
- 若后续补充 kcal 数据，再扩展热量统计。

## 7. 本地存储方案

存储方式：

- Zustand 管理业务状态
- AsyncStorage 持久化到本地
- App 离线可用，无需联网

建议数据结构：

- 用户设置：初始体重、目标体重、当前计划、计划开始日期
- 体重记录：日期、体重、备注
- 饮食记录：日期、计划、计划天数、是否按食谱完成、违规记录
- 运动记录：日期、爬坡完成状态、爬坡时长、力量完成状态、力量时长、备注
- 打卡汇总：日期、体重记录状态、饮食状态、运动状态、当日打卡状态
- 数据版本号：用于后续升级本地数据结构

持久化要求：

- 重启 App 后数据不丢失。
- 计划切换不删除历史记录。
- 本地重置必须二次确认。
- 后续如增加数据结构，需要提供版本迁移逻辑。

## 8. UI规范

- 圆角卡片
- 清晰文字层级
