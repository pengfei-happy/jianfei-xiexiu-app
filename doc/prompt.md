## 1. 总目标

实现“秦昊减脂打卡助手”第一版 App。

项目定位：面向大基数减脂用户的离线本地打卡 App，围绕秦昊本人原版 15 天减脂食谱和大冰 7 天循环减脂方案，提供饮食计划查看、每日打卡、体重记录、运动记录和数据看板能力。

技术栈规划：

- React Native + TypeScript
- Expo v56.0.0
- Tailwind RN
- Zustand 本地状态管理
- AsyncStorage 本地持久化
- 图表库用于周/月体重趋势展示

核心能力：

- 秦昊本人原版 15 天减脂食谱展示。
- 大冰 7 天循环减脂食谱展示。
- 每日体重、饮食、运动三类打卡。
- 连续打卡统计。
- 体重记录、累计减重、周/月趋势和历史记录。
- 五大底部 Tab：首页、食谱计划、数据看板、运动记录、我的。
- 全部数据本地持久化，离线可用。

## 3. 用户、计划与打卡规则

目标用户：大基数减脂用户。

体重设置：

- App 不预设初始体重。
- 用户首次使用时自行填写初始体重和目标体重。
- 体重单位固定为“斤”。
- 累计减重按用户真实体重记录计算，不按预估掉秤累加。
- 真实累计减重 = 初始体重 - 最新体重。
- 同一天多条体重记录，图表和当日值默认取当天最后一次记录。

默认计划：

- 首次进入 App 默认执行“秦昊 15 天计划”。
- 秦昊 15 天完成后，提示用户切换到“大冰 7 天循环计划”，不自动强制切换。
- 用户可在个人中心手动切换计划。
- 计划切换不删除历史记录。

打卡规则：

- 每天包含体重记录、饮食打卡、运动打卡三类动作。
- 任意完成一项，即视为当日打卡成功。
- 三项都完成时，视为当日完整打卡。
- 连续打卡天数按“当日是否打卡成功”统计。
- 若自然日未完成任何一项打卡，连续打卡天数清零。
- 支持补打卡，补打卡会影响对应日期的历史记录和连续打卡统计。
- 饮食打卡中，“未按食谱完成”仍算饮食记录已填写，但不计入饮食完成率。
- 运动打卡中，任意运动项填写或完成即有运动记录；爬坡快走和力量训练两项都完成才算运动完整完成。

数据展示口径：

- 只展示饮食和运动记录，不计算基础代谢。
- 热量缺口不做自动精确计算，除非后续确认摄入热量和运动消耗规则。
- 运动消耗热量数值暂标记为待确认或不展示。

## 7. UI 与体验规范

整体采用移动端优先的健康记录工具风格，信息密度适中，优先保证食谱内容、打卡状态和体重数据清晰可读。

视觉规范：

- 主色调：健康绿色，用于主按钮、选中状态、进度条和成功状态。
- 背景色：浅白或极浅灰，保证长文本阅读舒适。
- 警示色：用于禁忌、违规进食、重置数据和危险操作。
- 成功色：用于已打卡、已完成、连续打卡等正向反馈。
- 弱化色：用于待确认、空状态、历史说明和辅助信息。

尺寸规范：

- 页面水平安全边距：16。
- 页面区块垂直间距：16。
- 卡片内边距：16。
- 卡片之间间距：12。
- 列表项之间间距：8 到 12。
- 普通卡片圆角：8。
- 输入框、选择器、打卡卡片圆角：8。
- 小标签圆角：6。
- 页面标题：22，半粗或加粗。
- 区块标题：18，半粗。
- 卡片标题：16，半粗。
- 正文：14 到 16。
- 辅助说明：12 到 13。
- 数据大数字：24 到 32，按空间自适应。
- 长食谱文本默认 14 到 15，必须支持自动换行。

状态规范：

- 每个数据依赖视图都必须覆盖首次使用空状态。
- 未填写初始体重或目标体重要有空状态。
- 无体重记录、无运动记录、无历史打卡记录要有空状态。
- 待确认字段要清晰标记。
- AsyncStorage 读取中要有加载状态。
- 本地数据读取失败要有降级状态。
- 重置打卡、体重记录、全部本地数据都必须二次确认。

体验要求：

- 五大页面必须能通过底部 Tab 进入。
- 首页必须是默认首屏。
- 不要做营销落地页，首屏必须是可使用的 App 首页。
- 长食谱文本必须自动换行，不得溢出卡片。
- 底部 Tab 不得遮挡最后一个卡片。
- 页面在 iOS 和 Android 小屏设备上必须可滚动且内容不重叠。

## 8. TypeScript 数据接口规范

实现时必须建立统一类型文件，至少覆盖以下类型口径：

```ts
type PlanType = 'qin_hao_15' | 'dabing_7_cycle';

type CheckInStatus = 'none' | 'partial' | 'complete';

type MealType = 'breakfast' | 'lunch' | 'dinner' | 'snack';

type RecipePhase =
  | 'rapid_loss'
  | 'refeed_stabilize'
  | 'dabing_cycle';

interface RecipeMeal {
  type: MealType;
  label: string;
  content: string;
  isProvidedBySource: boolean;
}

interface RecipeDay {
  id: string;
  planType: PlanType;
  day: number;
  cycleDay?: number;
  phase: RecipePhase;
  stageName: string;
  estimatedWeightLossJin?: number;
  estimatedWeightLossText?: string;
  meals: RecipeMeal[];
  coreRule: string;
  warnings?: string[];
  principle?: string;
  sourceLocked: true;
}

interface WeightRecord {
  id: string;
  date: string;
  weightJin: number;
  note?: string;
  createdAt: string;
  updatedAt: string;
}

interface WeightSettings {
  initialWeightJin?: number;
  targetWeightJin?: number;
}

interface DietCheckIn {
  date: string;
  planType: PlanType;
  planDay: number;
  cycleDay?: number;
  hasDietRecord: boolean;
  followedRecipe: boolean;
  violationFoods: FoodViolation[];
  note?: string;
  updatedAt: string;
}

interface FoodViolation {
  id: string;
  date: string;
  foodName: string;
  note?: string;
}

interface DailyCheckIn {
  date: string;
  hasWeightRecord: boolean;
  hasDietRecord: boolean;
  hasExerciseRecord: boolean;
  status: CheckInStatus;
  isSuccessful: boolean;
  isComplete: boolean;
  updatedAt: string;
}

interface ExerciseRecord {
  id: string;
  date: string;
  inclineWalkCompleted: boolean;
  inclineWalkMinutes: number;
  strengthCompleted: boolean;
  strengthMinutes: number;
  hasExerciseRecord: boolean;
  isExerciseComplete: boolean;
  caloriesBurnedKcal?: number;
  note?: string;
  updatedAt: string;
}

interface PlanProgress {
  planType: PlanType;
  startDate: string;
  executionDay: number;
  qinHaoDay?: number;
  dabingCycleDay?: number;
  totalTargetLossJin?: number;
  currentLossJin?: number;
  progressRatio?: number;
}

interface UserSettings {
  currentPlan: PlanType;
  planStartDate: string;
  weightSettings: WeightSettings;
  dataVersion: number;
}
```

设计说明：

- `date` 使用本地自然日字符串，如 `YYYY-MM-DD`。
- 秦昊计划允许 `estimatedWeightLossJin`。
- 大冰计划不写每日数值，使用 `estimatedWeightLossText` 展示“稳步减脂、维稳塑形”类文案。
- `sourceLocked` 固定为 `true`，提醒后续实现不要提供编辑入口。
- `isSuccessful = hasWeightRecord || hasDietRecord || hasExerciseRecord`。
- `isComplete = hasWeightRecord && hasDietRecord && hasExerciseRecord`。

## 12. Zustand 状态模块与本地存储

### 12.1 recipeStore

状态：

- `currentPlan`
- `planStartDate`
- `qinHaoRecipes`
- `dabingRecipes`

动作：

- `setCurrentPlan(planType)`：切换当前计划。
- `setPlanStartDate(date)`：设置计划开始日期。
- `getExecutionDay(date)`：计算执行第几天。
- `getTodayRecipe(date)`：读取今日食谱。
- `getRecipeByDay(planType, day)`：读取指定食谱。

派生逻辑：

- 秦昊计划执行天数超过 15 天时，提示用户切换大冰计划，不强制切换。
- 大冰循环日 = `((执行天数 - 1) % 7) + 1`。
- 食谱数据只读，不提供修改动作。

### 12.2 weightStore

状态：

- `initialWeightJin`
- `targetWeightJin`
- `records`

动作：

- `setInitialWeight(weight)`
- `setTargetWeight(weight)`
- `addWeightRecord(record)`
- `updateWeightRecord(id, patch)`
- `getLatestWeight()`
- `getLatestWeightByDate(date)`

派生逻辑：

- 最新体重取最新创建或更新记录。
- 同日图表值取当天最后一次记录。
- 累计减重 = 初始体重 - 最新体重。
- 单日变化 = 当日体重 - 上一条有记录日期的体重。

### 12.3 checkInStore

状态：

- `dailyCheckIns`
- `dietCheckIns`
- `foodViolations`

动作：

- `syncWeightStatus(date)`
- `saveDietCheckIn(record)`
- `syncExerciseStatus(date)`
- `recalculateDailyStatus(date)`
- `recalculateStreak(today)`
- `getCompletionStats(range)`

派生逻辑：

- 当日成功 = 体重、饮食、运动任意一项有记录。
- 当日完整 = 体重、饮食、运动三项都有记录。
- 连续打卡从今天或指定日期向前按自然日倒推。
- 补打卡后重新计算受影响日期之后的连续打卡。
- 饮食完成率只统计 `followedRecipe = true` 的天数。

### 12.4 exerciseStore

状态：

- `records`
- 默认爬坡目标：60 分钟。
- 默认力量目标：30 分钟。

动作：

- `saveExerciseRecord(record)`
- `updateInclineWalk(date, completed, minutes)`
- `updateStrength(date, completed, minutes)`
- `updateExerciseNote(date, note)`
- `getExerciseRecordByDate(date)`

派生逻辑：

- 任意运动项填写或完成，即有运动记录。
- 爬坡完成且力量完成，才是运动完整完成。
- 当前版本不自动估算运动消耗热量。

### 12.5 AsyncStorage 键名

| 键名 | 内容 | 持久化策略 |
| --- | --- | --- |
| `slim_app:user_settings:v1` | 当前计划、计划开始日期、初始体重、目标体重、数据版本 | 永久保存，重置全部数据时清除 |
| `slim_app:weight_records:v1` | 体重历史记录 | 永久保存，可单独重置 |
| `slim_app:diet_checkins:v1` | 饮食打卡和违规进食记录 | 永久保存，可随打卡重置 |
| `slim_app:exercise_records:v1` | 运动记录 | 永久保存，可随打卡重置 |
| `slim_app:daily_checkins:v1` | 每日打卡汇总 | 永久保存，可随打卡重置 |
| `slim_app:data_version` | 本地数据结构版本 | 永久保存，用于后续迁移 |

保存原则：

## 16. 如果遇到不明确问题

整个实现过程默认不会有人类参与。遇到不明确问题时，按以下优先级自行决策：

1. 以 `doc/proposal.md` 为最高业务依据。
2. 以 `doc/detailed-design.md` 为最高设计与技术依据。
3. 以 `doc/tasks/*.md` 为任务边界依据。
4. 以 Expo v56.0.0 官方文档和当前项目依赖版本为技术依据。
5. 如文档明确禁止编造，则宁可展示“待确认”或不展示，也不要推测。

只有当需求文档、详细设计和任务文件互相冲突，且无法通过上述优先级消解时，才允许暂停并提出问题。除此之外，不要等待人工确认。
