# 秦昊减脂打卡助手概要设计

## 1. 全局设计原则

### 1.1 组件复用优先

App 采用页面容器与业务组件分离的设计方式。页面负责路由入口、布局组织和数据装配，公共组件负责稳定的展示与交互能力。

核心复用原则：

- 数据展示统一使用数据卡片组件承载，避免各页面重复实现统计卡片样式。
- 计划进度、减重进度、打卡完成度统一使用进度条组件。
- 日期切换、历史查询、补打卡入口统一使用日期选择器。
- 饮食、运动、体重打卡状态统一抽象为打卡复选卡片。
- 体重趋势统一使用折线图表组件。
- 五大页面统一挂载到底部 Tab 导航。
- 秦昊15天食谱与大冰7天循环食谱统一使用食谱详情卡片展示。

### 1.2 TypeScript 强类型

项目使用 TypeScript 严格模式约束核心数据结构，食谱、体重记录、饮食记录、运动记录、打卡汇总和用户设置均需要定义明确类型。

强类型目标：

- 避免计划类型、计划天数、循环天数等字段混用。
- 避免食谱字段缺失时被错误渲染。
- 保证 Zustand store 的输入、输出和计算选择器具备稳定类型。
- 为后续单元测试、数据迁移和本地持久化提供稳定契约。

### 1.3 离线优先

App 第一版按离线本地应用设计，无需联网即可查看食谱、记录体重、完成打卡、查看历史数据和数据看板。

离线设计要求：

- 秦昊15天食谱与大冰7天循环食谱以内置静态 JSON 形式随 App 发布。
- 用户产生的数据全部写入本地 AsyncStorage。
- 重启 App 后状态从本地持久化数据恢复。
- 当前版本不依赖服务端、不设计登录注册、不设计云同步。

### 1.4 食谱数据只读

食谱数据是业务事实源，必须严格来自 `doc/proposal.md` 中确认的原文餐单规则。

数据只读原则：

- App 不允许用户编辑内置食谱。
- 不在代码或设计中新增、删减、替换、估算每日三餐内容。
- 大冰计划不估算每日掉秤数字。
- 未确认的热量、运动消耗、健康风险提示等信息只保留扩展口，不进入自动计算。

### 1.5 阶段工程约束

后续实现阶段遵循以下约束：

- 每个模块任务完成后单独提交代码，分阶段保存变更。
- 开发过程中同步更新 `doc/tasks/progress.md` 记录完成状态。
- 业务逻辑、状态计算、食谱解析模块必须配套单元测试。
- 页面交互增加基础自测逻辑，不遗留无测试源码。
- 多 Agent 总提示词、各阶段 Prompt、设计文档统一归入 `doc` 目录。
- 使用 context-management 记录全流程阶段决策，留存需求与设计变更记录。

## 2. 整体技术分层架构

系统分为页面路由层、业务组件层、食谱数据源层、全局状态存储层和工具函数层。

```text
页面路由层
  HomeScreen 首页
  RecipePlanScreen 食谱计划
  DashboardScreen 数据看板
  ExerciseScreen 运动记录
  ProfileScreen 个人中心
        |
        v
业务组件层
  DataCard 数据卡片
  ProgressBar 进度条
  DatePicker 日期选择器
  CheckInCard 打卡复选卡片
  LineChart 折线图表
  BottomTabNavigator 底部Tab导航
  RecipeDetailCard 食谱详情卡片
        |
        v
全局状态存储层
  weightStore 体重store
  checkInStore 打卡store
  recipeStore 食谱store
  exerciseStore 运动store
        |
        v
工具函数层
  日期计算
  计划天数计算
  连续打卡计算
  体重趋势计算
  完成率计算
  本地持久化适配
        |
        v
食谱数据源层
  qinHao15DayRecipes.json
  dabing7DayRecipes.json
```

### 2.1 页面路由层

页面路由层负责五大页面的导航组织、页面级状态订阅和组件编排。

职责边界：

- 只处理页面级布局和用户操作入口。
- 不直接硬编码食谱业务规则。
- 不直接操作 AsyncStorage。
- 不重复实现跨页面可复用组件。

### 2.2 业务组件层

业务组件层负责复用 UI 与局部交互。

职责边界：

- 接收 props 展示数据和触发回调。
- 不直接持久化数据。
- 不修改只读食谱数据。
- 对长食谱文本、空状态、待确认字段提供稳定展示。

### 2.3 食谱数据源层

食谱数据源层提供秦昊15天计划和大冰7天循环计划的静态只读数据。

职责边界：

- 仅存放结构化静态 JSON 数据。
- 秦昊15天包含 Day 1 到 Day 15 的阶段、三餐、规则、预计掉秤。
- 大冰7天包含循环第1天到第7天的三餐、原则和循环规则。
- 不存放用户打卡、体重、运动记录。

### 2.4 全局状态存储层

全局状态存储层使用 Zustand 按业务域拆分 store，并通过 AsyncStorage 进行本地持久化。

模块边界：

- `weightStore`：用户体重设置、体重记录、最新体重、累计减重。
- `checkInStore`：饮食打卡、体重记录状态、运动记录状态、每日打卡汇总、连续打卡。
- `recipeStore`：当前计划、计划开始日期、当前计划日、食谱读取选择器。
- `exerciseStore`：运动记录、默认运动项目、运动历史。

### 2.5 工具函数层

工具函数层承载可测试的纯计算逻辑。

核心函数范围：

- 计算秦昊计划第几天。
- 计算大冰循环第几天：`((执行天数 - 1) % 7) + 1`。
- 计算单日打卡状态：`none`、`partial`、`complete`。
- 计算连续打卡天数。
- 计算累计减重：`初始体重 - 最新体重`。
- 计算计划完成率、饮食完成率、运动完成率。
- 生成周视图、月视图体重趋势数据。
- 处理同一天多次体重记录时取最后一次记录。

## 3. 页面路由清单与各页面职责

### 3.1 首页

首页是每日执行入口，展示当天最关键的计划、食谱、打卡和进度信息。

页面职责：

- 展示当前执行计划标签：秦昊15天计划或大冰7天循环计划。
- 展示当日日期。
- 展示今日预估掉秤信息。
- 展示体重记录、饮食打卡、运动打卡三类入口。
- 展示当日打卡状态与连续打卡天数。
- 展示今日早餐、午餐、晚餐完整食谱预览。
- 展示今日核心规则、进食禁忌或注意事项。
- 展示秦昊15天总预估掉秤进度。

组件复用：

- `DataCard` 展示今日计划、连续打卡、累计减重等摘要信息。
- `CheckInCard` 展示三类打卡入口。
- `ProgressBar` 展示真实累计减重相对 11.3 斤目标的完成度。
- `RecipeDetailCard` 展示今日食谱预览。

### 3.2 食谱计划

食谱计划页用于完整浏览秦昊15天食谱和大冰7天循环食谱。

页面职责：

- 提供秦昊15天计划与大冰7天循环计划的双标签切换。
- 秦昊计划按 Day 1 到 Day 15 展示。
- 大冰计划按循环第1天到第7天展示。
- 点击某一天展开完整详情。
- 同一时间只展开一个天数，避免移动端信息过载。
- 在每日详情中展示饮食打卡状态和违规进食记录入口。

组件复用：

- `RecipeDetailCard` 作为每日食谱详情的统一展示组件。
- `CheckInCard` 用于展示“今日是否按食谱完成饮食”。
- `DatePicker` 可用于跳转真实日期对应的计划日。

### 3.3 数据看板

数据看板页用于展示真实体重、打卡状态、趋势图和历史记录。

页面职责：

- 展示今日饮食打卡状态。
- 展示今日运动打卡状态。
- 展示连续打卡天数。
- 展示累计减重。
- 展示计划完成率。
- 展示最近7天和最近30天体重趋势。
- 展示历史每日掉秤记录清单。
- 对摄入热量、运动消耗、热量缺口保持不展示或标记待确认。

组件复用：

- `DataCard` 展示核心指标卡片。
- `LineChart` 展示体重趋势折线图。
- `ProgressBar` 展示计划完成率、饮食完成率和运动完成率。
- `DatePicker` 用于历史记录筛选。

### 3.4 运动记录

运动记录页用于记录每日爬坡快走与力量训练情况。

页面职责：

- 展示默认运动项目：爬坡快走1小时、力量训练30分钟。
- 支持记录爬坡快走是否完成。
- 支持记录爬坡快走时长。
- 支持记录力量训练是否完成。
- 支持记录力量训练时长。
- 支持填写当日运动备注。
- 展示运动历史。
- 展示运动提示，不计算运动消耗 kcal。

组件复用：

- `CheckInCard` 展示爬坡快走和力量训练完成状态。
- `DataCard` 展示运动提示和历史摘要。
- `DatePicker` 用于切换记录日期。

### 3.5 个人中心

个人中心页用于用户设置、计划切换、须知查看和本地数据管理。

页面职责：

- 设置初始体重。
- 设置目标体重。
- 切换当前计划。
- 展示秦昊15天计划和大冰7天循环计划切换入口。
- 展示减脂须知和嘴馋解馋法则。
- 查看历史打卡记录。
- 重置打卡和体重记录。
- 重置全部本地数据。
- 所有重置操作必须二次确认。

组件复用：

- `DataCard` 展示用户设置、计划信息和须知内容。
- `ProgressBar` 展示目标体重进度。
- `DatePicker` 用于历史记录查看。

## 5. 全局状态模块划分、数据流向

### 5.1 Zustand Store 模块

#### weightStore

管理体重相关状态。

核心数据：

- 初始体重。
- 目标体重。
- 体重记录列表。
- 最新体重。
- 累计减重。

核心动作：

- 设置初始体重。
- 设置目标体重。
- 新增或更新某日体重记录。
- 获取某日最后一次体重记录。
- 计算累计减重和单日变化。

#### checkInStore

管理每日打卡汇总与饮食打卡状态。

核心数据：

- 每日打卡汇总。
- 饮食打卡记录。
- 违规进食记录。
- 连续打卡天数。

核心动作：

- 写入体重记录状态。
- 写入饮食打卡状态。
- 写入运动记录状态。
- 重新计算当日打卡状态。
- 补打卡后重新计算连续打卡天数。
- 计算饮食完成率和总打卡完成率。

#### recipeStore

管理当前计划与食谱读取。

核心数据：

- 当前计划：秦昊15天计划或大冰7天循环计划。
- 计划开始日期。
- 当前执行天数。
- 当前秦昊计划日。
- 当前大冰循环日。

核心动作：

- 切换计划。
- 设置计划开始日期。
- 读取今日食谱。
- 读取指定计划日食谱。
- 判断秦昊15天是否完成并提供切换提示。

#### exerciseStore

管理运动记录。

核心数据：

- 运动记录列表。
- 爬坡快走完成状态。
- 爬坡快走时长。
- 力量训练完成状态。
- 力量训练时长。
- 当日运动备注。

核心动作：

- 新增或更新某日运动记录。
- 判断当天是否存在运动记录。
- 判断当天运动是否完整完成。
- 读取运动历史。

### 5.2 全局数据流向

```text
用户操作
  -> 页面路由层接收事件
  -> 调用对应 Zustand store action
  -> store 更新内存状态
  -> AsyncStorage 持久化用户数据
  -> selector 重新计算派生数据
  -> 页面和公共组件重新渲染
```

食谱读取数据流：

```text
页面请求今日食谱
  -> recipeStore 根据当前计划和执行天数计算计划日
  -> 读取内置只读 JSON
  -> 返回结构化食谱数据
  -> RecipeDetailCard 渲染完整三餐和规则
```

打卡汇总数据流：

```text
体重记录 / 饮食打卡 / 运动记录
  -> 对应 store 写入业务记录
  -> checkInStore 同步更新当日三类状态
  -> 计算 none / partial / complete
  -> 重新计算连续打卡天数和完成率
  -> 首页与数据看板同步展示
```

体重趋势数据流：

```text
weightStore 体重记录
  -> 工具函数按日期聚合
  -> 同日多次记录取最后一次
  -> 生成最近7天或最近30天序列
  -> LineChart 渲染折线图
```

### 5.3 持久化边界

需要持久化的数据：

- 用户设置。
- 体重记录。
- 饮食记录。
- 运动记录。
- 打卡汇总。
- 当前计划和计划开始日期。
- 数据版本号。

不需要持久化的数据：

- 内置食谱静态数据。
- 页面临时展开状态。
- 图表临时视图模式。
- 弹窗打开关闭状态。

## 6. 内置食谱静态数据存储方案

### 6.1 数据文件划分

食谱数据建议拆分为两个只读 JSON 文件：

```text
src/data/recipes/qinHao15DayRecipes.json
src/data/recipes/dabing7DayRecipes.json
```

以上为后续实现阶段建议路径，本阶段不创建代码或数据文件。

### 6.2 秦昊15天食谱结构

秦昊15天计划按固定15天存储。

建议字段：

- `day`：计划第几天，1 到 15。
- `stage`：阶段名称。
- `phase`：极速减脂期或复食巩固期。
- `estimatedWeightLossJin`：当日预计掉秤，单位斤。
- `breakfast`：早餐原文。
- `lunch`：午餐原文。
- `dinner`：晚餐原文。
- `snack`：加餐，原文未提供则为空或不展示。
- `rules`：核心规则。
- `warnings`：禁忌或注意事项。

约束：

- 15天总预估掉秤固定为 11.3 斤。
- 每日预计掉秤必须来自 `doc/proposal.md`。
- 餐单展示以完整原文内容为准，不做摘要替代。

### 6.3 大冰7天循环食谱结构

大冰7天循环计划按固定7天存储。

建议字段：

- `cycleDay`：循环第几天，1 到 7。
- `stage`：原文阶段。
- `breakfast`：早餐原文。
- `lunch`：午餐原文。
- `dinner`：晚餐原文。
- `principle`：原则。
- `rules`：通用规则。

约束：

- 第7天结束后下一天重新从第1天开始。
- 历史记录保存真实日期和循环第几天。
- 不提供每日预计掉秤数字。
- 今日预估掉秤展示文案为“大冰循环以稳步减脂、维稳塑形为主”。

### 6.4 食谱数据访问规则

食谱数据访问通过 `recipeStore` 或食谱选择器完成，页面不直接关心 JSON 文件结构。

访问方式：

- `getTodayRecipe()`：根据当前计划和日期返回今日食谱。
- `getRecipeByPlanDay(plan, day)`：返回指定计划日食谱。
- `getCurrentPlanDay(date)`：计算秦昊计划第几天。
- `getDabingCycleDay(executionDay)`：计算大冰循环第几天。

| 状态管理 | `zustand` | 多模块全局状态管理 |

### 7.1 依赖使用边界

- 路由依赖只负责页面切换和底部 Tab，不承载业务状态。
- 图表依赖只负责展示体重趋势，不负责数据计算。
- AsyncStorage 只保存用户本地数据，不保存只读食谱。
- Zustand 负责业务状态与派生选择器，不在组件中散落重复计算。
- TypeScript 严格模式用于约束核心业务模型，不以 `any` 绕过食谱和记录字段。

## 8. 架构边界与后续扩展点

### 8.1 当前版本明确不做

- 不联网。
- 不登录注册。
- 不云同步。
- 不导出 CSV 或 JSON 文件。
- 不自动计算摄入热量。
- 不自动计算运动消耗热量。
- 不自动计算热量缺口。
- 不修改或扩写原始食谱内容。

### 8.2 后续可扩展点

- 增加本地数据结构版本迁移。
- 增加每日提醒或通知。
- 增加健康风险提示或免责声明弹窗。
- 增加深色模式。
- 增加历史数据导出。
- 在补充 kcal 规则后扩展热量统计模块。
